## 二、技术架构

### 2.1 技术栈（零依赖纯前端 + Serverless 后端）

| 层 | 技术 | 角色 |
|---|------|------|
| 前端 | 原生 HTML/CSS/JavaScript | 无框架，单页应用 |
| 数据 | Supabase (PostgreSQL + REST API) | 云端存储题库/类型/留言 |
| 部署 | Cloudflare Workers (Static Assets) | 全球 CDN，免费 |
| 域名 | `11-woti.icu`（阿里云购买，1 元/年） | 自定义域名解决国内访问 |
| 命令行 | Wrangler CLI + Node.js | 一键部署 `npx wrangler deploy` |

### 2.2 项目结构

```
woti/
├── index.html              # 主页面（首页/答题/结果/分享/管理面板/留言墙）
├── css/
│   └── style.css           # 暖白+军绿浅色主题
├── js/
│   ├── app.js              # 核心逻辑（答题引擎、结果计算、管理面板、Canvas 分享）
│   └── supabase.js         # Supabase REST API 封装（无 SDK 依赖）
├── data/
│   ├── questions.json      # 兜底题库（Supabase 失败时使用）
│   └── types.json          # 兜底类型数据
├── wrangler.toml           # Cloudflare 部署配置
├── supabase_setup.sql      # 数据库初始化 SQL
└── .assetsignore           # 排除 .git/.wrangler 不上传
```

### 2.3 数据流

```
用户访问 → app.js 启动 → 优先读 Supabase
                              ├─ 成功 → 渲染页面
                              └─ 失败 → 回退本地 JSON
管理员编辑 → 写入 Supabase → 所有访客实时看到最新内容
```

| 表名 | 用途 | RLS 策略 |

## 四、关键问题与复盘

### 4.1 ⚠️ Cloudflare Git 部署失败（白折腾一圈）

**症状：** 通过 Cloudflare 面板连接 GitHub 部署成功，但 URL 完全打不开（`ERR_CONNECTION_TIMED_OUT`）。

**当时的错误推断：** 以为是 Cloudflare Git CI 没正确处理 `[assets]` 配置。

**真实原因：** **`workers.dev` 域名在国内被 DNS 污染**。即使开 VPN，从国内访问 `*.workers.dev` 也大概率超时。Git 部署其实是成功的，只是域名访问不通。

**教训：**
- 调试要先验证假设，而不是直接换方案
- 国内做静态网站，**workers.dev 默认域名不可用**，必须绑自定义域名

### 4.2 解决方案：买域名 + Cloudflare DNS

1. 阿里云买 `.icu` 域名（1 元/年，不需要备案）
2. 把域名加到 Cloudflare → 拿到 2 个 NS 服务器
3. 阿里云改 DNS 服务器为 Cloudflare 的
4. 在 `wrangler.toml` 添加 `routes` 绑定到 Worker
5. 国内即可正常访问

### 4.3 Wrangler CLI 认证问题

**症状：** `wrangler login` 在 Claude Code 里报 `spawn UNKNOWN`，无法打开浏览器。

**解决：** 用 **API Token** 替代 OAuth：
- Cloudflare → Profile → API Tokens → Create Token
- 选 "Edit Cloudflare Workers" 模板
- 用环境变量 `CLOUDFLARE_API_TOKEN` 注入

### 4.4 Supabase REST API 的坑

| 问题 | 解决 |
|------|------|
| PATCH 返回 204 时 `res.json()` 崩溃 | 加判断：`if (res.status === 204) return null` |
| `Prefer: return=minimal` 响应空 | 加 `if (!text) return null` |
| Bulk insert 时所有对象 keys 必须一致 | 给 hidden 类型也补全所有字段 |
| RLS 默认禁止所有操作 | 显式建 SELECT/INSERT/UPDATE/DELETE policy |

### 4.5 字符编码踩坑

阿里云域名首年 1 元活动，但 `.icu` 没参加，最终买的 `11-woti.icu` 略贵但可用。

Shell 里 `curl` 的中文 JSON 容易被编码搞坏，最终用 Node.js 的 fetch 替代调试。

## 八、技术决策回顾

### 为什么不用框架（React/Vue）？
- 项目规模小，纯静态展示 + 简单交互，原生 JS 完全够
- 零依赖意味着部署简单、加载快、出问题好排查
- 参考项目 HERTI/SBTI 也都是原生 JS

### 为什么选 Supabase 不选 Firebase？
- Firebase 在国内访问差
- Supabase 用 PostgreSQL（更熟悉的 SQL），有完善的 REST API
- 免费额度足够（500MB 数据库，50K 月活用户）

### 为什么选 Cloudflare Workers 不选 Pages？
- 实际上 Cloudflare 已经把 Pages 合并到 Workers 里了，新建项目走的就是 Workers
- 静态资源用 `[assets]` 配置，效果等同于 Pages

### 为什么不用 Vercel/Netlify？
- 国内访问 Vercel 也慢
- Cloudflare 配自定义域名后效果更好
- Cloudflare 免费版无带宽限制
